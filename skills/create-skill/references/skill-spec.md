# Skill 结构规范

## 目录结构

一个标准的 skill 目录结构如下：

```text
skills/<skill-name>/
├── README.md               # 必需 — skill 说明文档
├── SKILL.md                # 必需 — skill 定义文件
└── references/             # 可选 — 参考资料目录
    ├── <reference-1>.md
    └── <reference-2>.md
```

## SKILL.md 文件结构

### Frontmatter（YAML 头部）

| 字段 | 必填 | 说明 | 示例 |
| ---- | ---- | ---- | ---- |
| `name` | 是 | skill 唯一标识，小写英文+连字符 | `process-requirement` |
| `description` | 是 | 触发描述，包含触发词和激活条件 | 见下方示例 |
| `metadata.author` | 否 | 作者名称 | `"张三"` |
| `metadata.version` | 否 | 语义化版本号 | `"1.0"` |

#### name 命名规则

- 全部小写英文
- 单词间用连字符（`-`）连接
- 简洁且有描述性
- 例：`process-requirement`、`create-skill`、`git-commit`

#### description 编写规则

- 使用 `>-` 多行折叠格式
- 第一句话概述功能
- 然后列出触发词（中英文都要覆盖）
- 最后说明 skill 的核心价值

示例：

```yaml
description: >-
  在编码之前引导结构化的需求分析流程。当用户说"新需求"、"开发功能"、"实现功能"、
  "需求分析"、"开始开发"、"帮我做"、"帮我写"、"加个功能"、"改一下"、
  "process requirement"、"analyze requirement"、"new feature"、
  "implement feature"等涉及构建或修改功能的请求时激活此技能。
  此技能确保在任何代码变更之前完成充分的需求沟通、影响分析和用户确认。
```

### 正文结构

正文使用 Markdown 格式，推荐以下结构：

#### 1. 概述（必需）

- 用 `## 概述` 标题
- 1-2 句话定义 Agent 的角色
- 明确核心职责和底线约束

示例：

```markdown
## 概述

你是一位资深软件架构师。在编写任何代码之前，你必须引导用户完成结构化的需求分析流程。
绝对不要直接跳到编码环节。
```

#### 2. 工作流程（必需）

- 用 `## 工作流程` 标题
- 用 `### 第N阶段：阶段名` 分阶段
- 每阶段包含：
  - 触发条件 / 前置条件
  - 具体步骤（编号列表）
  - 输出格式（用代码块包裹）
  - 进入下一阶段的条件

关键原则：

- 需要用户确认的节点必须显式标注
- 输出格式要具体，用代码块让 Agent 容易遵循
- 步骤描述要可操作，避免模糊指令

#### 3. 规则（推荐）

- 用 `## 规则` 标题
- 列表形式，每条规则独立
- 包含"必须做"和"绝对不能做"两类

示例：

```markdown
## 规则

- **绝对不能跳过第一阶段或第二阶段。** 每个阶段都必须获得用户确认才能继续。
- 使用用户所用的语言进行沟通。
- 如果用户的需求描述模糊，主动提问，不要自行假设。
```

## README.md 文件规范

每个 skill 必须有独立的 README.md，作为用户了解该 skill 的入口。

### 必需章节

| 章节 | 标题级别 | 说明 |
| ---- | -------- | ---- |
| skill 名称 | `#` | 一级标题，使用 skill 的 `name` |
| 功能描述 | 正文 | 标题下一句话概述 |
| 功能特性 | `##` | 列表形式列出核心能力 |
| 执行流程 | `##` | 按阶段用 `###` 子标题描述步骤 |
| 安装 | `##` | 包含 `npx skills add ... --skill <name>` 命令 |
| 使用方式 | `##` | 列出触发词，中英文分组 |
| 文件结构 | `##` | 用 `text` 代码块展示目录树 |

### 编写要求

- 功能特性用无序列表，每项一句话
- 执行流程只需简要描述各阶段的步骤，不需要输出格式模板（那是 SKILL.md 的职责）
- 安装命令使用 `bash` 代码块
- 文件结构使用 `text` 代码块，用树形缩进展示
- 语言风格与 SKILL.md 保持一致

## 参考文件规范

### 何时需要参考文件

- skill 需要领域特定的模板或提示词
- 工作流程中需要查阅的分类清单
- 通用检查项或评估标准

### 格式要求

- 文件名：小写英文+连字符，`.md` 后缀
- 格式：Markdown
- 用二级标题（`##`）按类别分节
- 内容用列表或表格组织，便于 Agent 快速定位
- 避免大段叙述性文字

## 触发词设计指南

### 覆盖面

- 中文口语化表达（"帮我做"、"来一个"）
- 中文正式表达（"需求分析"、"创建功能"）
- 英文表达（"create"、"implement"、"new feature"）

### 避免冲突

- 检查已有 skill 的触发词，避免重叠
- 触发词要足够具体，减少误触发
- 如有交叉，在 description 中说明边界

### 反面触发词

在 description 中可以说明不应触发的场景：

```text
不应在以下场景触发：纯代码问答、查看文件内容、简单的 bug 修复
```
